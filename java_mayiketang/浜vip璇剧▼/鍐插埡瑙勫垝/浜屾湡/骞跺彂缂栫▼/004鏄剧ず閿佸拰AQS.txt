2.ReentrantLock
    锁的可重入：同一个线程如果已经获取了锁，可以多次继续申请到锁的使用权。
    公平锁与非公平锁：先到的先执行，后到的慢执行叫做公平锁，反之叫做非公平锁；
                     非公平锁的性能比公平锁的性能高，因为使用公平锁的时候，cpu在恢复一个被挂起
                     的线程与该线程真正开始运行之间存在着严重的延时问题。

5.LockSupport

    LockSupport 增加了 park(Object blocker)、parkNanos(Object blocker,long nanos)和
    parkUntil(Object blocker,long deadline)3 个方法，用于实现阻塞当前线程的功能，
    其中参数 blocker 是用来标识当前线程在等待的对象（以下称为阻塞对象），该对象主要用于问题排查和系统监控

7.AbstractQueuedSynchronizer(同步器)
    AQS使用的方式主要是继承，子类通过继承AQS并实现抽象方法来管理同步，在AQS中有一个int类型的state，当state为1时
    表示获取到了锁，为0时表示释放了锁，提供了getState()、setState(int newState)和
    compareAndSetState(int expect,int update)来保证state共享变量的安全问题。
    在实现上，子类推荐定义，在子类中自定义同步组件的静态内部类；AQS自身没有实现任何同步接口，它仅仅是定义了若干同步
    状态获取和释放的方法来供自定义同步组件使用，既可以支持独占的获取同步状态，也可以支持共享式地获取同步状态。
    模板方法：
        同步器的设计是基于模板方法模式，定义一个操作中的算法骨架，而将一些步骤的实现延迟到子类中。
        模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。
        我们最常见的就是 Spring 框架里的各种 Template。
8.AQS中的方法
    模板方法：实现自定义组件的，将会调用同步器的模板方法
        void acquire(int arg)   独占试获取同步状态，如果成功就返回，否则，将进入同步队列等待，该方法将会调用重写的
                                tryAcquire(int arg)方法
        void acquireInterruptibly(int arg)  与acquire方法相同，但是该方法会判断是否中断，如果当前线程被中断，
                                            会抛出InterruptedException并返回
        boolean tryAcquireNanos(int arg,long nanos) 在acquireInterruptibly增加了限时操作，如果在设置的超时时间内，
                                                    没有获取到同步状态，返回false，否则返回true
        void acquireShared(int arg) 共享式获取同步状态，未获取到同步状态，则会进入同步队列等待，与独占式的区别是
                                    同一时刻可以多个线程获取到同步状态
        void acquireSharedInterruptibly(int arg) 与acquireInterruptibly相同，但是会响应式中断，也就是会判断是否中断
        boolean tryAcquireSharedNanos(int arg,long nanos) 在tryAcquireNanos上增加了超时机制
        boolean release(int arg) 独占式释放同步，释放后，将同步队列的第一个节点的线程唤醒
        boolean releaseShared(int arg) 共享式释放同步
        Collection<Thread> getQueuedThread() 获取同步队列上的线程集合
   可重写的方法：
        protected boolean tryAcquire(int arg)：独占式同步，执行前需要判断当前状态是否符合预期，然后再进行CAS同步状态
        protected boolean tryRelease(int arg)：独占式释放同步，等待获取同步状态的线程有机会获取同步状态
        protected int tryAcquireShared(int arg)：共享式获取同步，返回大于0的值表示获取成功，反之失败
        protected boolean tryReleaseShared(int arg)：共享式释放同步状态
        protected boolean isHeldExclusively()：当前同步器是否是独占模式下被线程占用，表示是否被当前线程占用
   访问或修改同步状态的方法：
    getState()：获取当前同步状态。
    setState(int newState)：设置当前同步状态。
    compareAndSetState(int expect,int update)：使用 CAS 设置当前状态，该方法能够保证状态设置的原子性
9.AQS源码
    节点Node：
        Node是AQS中的内部类
    两种等待模式：
        SHARED：表示线程以共享的模式等待锁（如 ReadLock）
        EXCLUSIVE：表示线程以互斥的模式等待锁（如 ReentrantLock），互斥就是一把锁只能由一个线程持有，
                   不能同时存在多个线程使用同一个锁
    线程在队列中的状态枚举：
        CANCELLED：值为 1，表示线程的获锁请求已经“取消
        SIGNAL：值为-1，表示该线程一切都准备好了,就等待锁空闲出来给我，这是它的下一个节点去设置的，叫醒的时候会去判断
                这个状态，根据这个状态去判断是否需要去叫醒下一个节点；
        CONDITION：值为-2，表示线程等待某一个条件（Condition）被满足，等待满足条件后将节点加入到同步队列中；
        PROPAGATE：值为-3，当线程处在“SHARED”模式时，该字段才会被使用上初始化
        Node 对象时，默认为 0
    成员变量：
        waitStatus：该int变量表示线程在队列中的状态，其值就是上述提到的CANCELLED、SIGNAL、CONDITION、PROPAGATE
        prev：该变量类型为Node 对象，表示该节点的前一个Node 节点（前驱）
        next：该变量类型为Node 对象，表示该节点的后一个Node 节点（后继）
        thread：该变量类型为Thread 对象，表示该节点的代表的线程
        nextWaiter：该变量类型为Node对象，表示等待condition条件的Node节点
    当前线程获取同步状态失败时，同步器会将当前线程以及等待状态等信息构造成为一个节点（Node）并将其加入同步队列，
    同时会阻塞当前线程，当同步状态释放时，会把首节点中的线程唤醒，使其再次尝试获取同步状态。同步队列中的节点（Node）
    用来保存获取同步状态失败的线程引用、等待状态以及前驱和后继节点。
    head 和 tail：
        AQS 还拥有首节点（head）和尾节点（tail）两个引用，一个指向队列头节点，而另一个指向队列尾节点。
        注意：因为首节点 head 是不保存线程信息的节点，仅仅是因为数据结构设计上的需要，在数据结构上，
              这种做法往往叫做“空头节点链表”。对应的就有 “非空头结点链表”；
              这里的意思就是头节点的thread是为空的；



1.AQS同步队列中的增加和移除
    节点加入到同步队列
        当一个线程获取了锁，其它线程获取不了锁，也就是获取锁失败，AQS会将这个线程构造成一个节点(Node)，并将其加入到
        队列的尾部，而必须保证尾部的线程安全，AQS提供了一个基于CAS的设置尾节点的方法：compareAndSetTail(Node Expect,
        Node Update)就是说当前节点认为的尾节点和内存中的尾节点是一致才会返回true，当前节点才会设置成尾节点。
    首节点的变化：
        首节点是获取同步状态成功的节点，首节点的线程在释放同步状态的同时，会唤醒后继节点，而后继节点在获取同步状态成功
        后将自己设置为首节点；设置首节点是获取同步状态的线程来完成的，而只能有一个线程获取同步状态成功，所以设置成首节
        点不需要使用CAS来保证线程安全，它只需要将自己设置成首节点，并且将原首节点的next引用断开即可。
2.独占式同步状态与释放锁
   获取：
        通过AQS的acquire(int tag)方法获取同步状态，主要完成同步状态的获取、节点构建、加入同步队列以及自旋等待
        public final void acquire(int arg) {
                if (!tryAcquire(arg) &&
                    acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
                    selfInterrupt();
        }
        tryAcquire(arg)如果同步状态获取失败返回false，则构建同步节点(独占式 Node.EXCLUSIVE)，然后执行addWaiter方法，
        将该队列加入到同步队列的尾部，最后通过acquireQueued是该节点进入死循环的方式获取同步状态，如果获取不到则阻塞节点
        中的线程，而阻塞的节点则主要依靠前驱结点的出队或阻塞线程被中断来实现。
        final boolean acquireQueued(final Node node, int arg) {
                boolean failed = true;
                try {
                    boolean interrupted = false;
                    for (;;) {
                        final Node p = node.predecessor();
                        if (p == head && tryAcquire(arg)) {
                            setHead(node);
                            p.next = null; // help GC
                            failed = false;
                            return interrupted;
                        }
                        if (shouldParkAfterFailedAcquire(p, node) &&
                            parkAndCheckInterrupt())
                            interrupted = true;
                    }
                } finally {
                    if (failed)
                        cancelAcquire(node);
                }
         }
        acquireQueued方法中，如果当前节点的父节点为头节点，则调用一次tryAcquire获取同步状态，如果没有获取成功，进入
        shouldParkAfterFailedAcquire检查是否需要阻塞返回true，进入parkAndCheckInterrupt方法中调用LockSupport进行阻塞，
        这里为什么需要这样设计进行阻塞呢，因为当一个节点获取或成功使用完释放后，会调用release方法释放锁，在释放锁的时候
        会唤醒后继节点。
        public final boolean release(int arg) {
                if (tryRelease(arg)) {
                    Node h = head;
                    if (h != null && h.waitStatus != 0)
                        unparkSuccessor(h);
                    return true;
                }
                return false;
        }
        tryRelease释放锁后成功后，调用unparkSuccessor方法该方法中使用LockSupport来唤醒后继节点，而在unparkSuccessor
        方法中，如下代码：
        Node s = node.next;
        if (s == null || s.waitStatus > 0) {
            s = null;
            for (Node t = tail; t != null && t != node; t = t.prev)
                if (t.waitStatus <= 0)
                    s = t;
        }
        if (s != null)
            LockSupport.unpark(s.thread);
        这段代码的意思是，被唤醒的后继节点处于一个cancel状态，（我推测开发者的思路这样的：后继节点处于被 cancel 状态，
        意味着当锁竞争激烈时，队列的第一个节点等了很久（一直被还未加入队列的节点抢走锁）包括后续的节点 cancel的几率都
        比较大，所以）先从尾开始遍历，找到最前面且没有被cancel的节点。
   总结：
        在获取同步状态时，AQS同步器维护了一个同步队列，获取失败后会加入到这个队列并在队列中进行自旋：移除队列（或者说
        停止自旋）的条件是前驱节点为头且获取到了同步状态，在释放时使用tryRelease(int arg)方法释放同步状态，然后唤醒
        head 指向节点的后继节点。
3.共享同步锁获取
    在acquireShared(int arg)方法中，同步器调用tryAcquireShared(int arg)方法尝试获取同步状态tryAcquireShared(int arg)
    方法返回值为int类型，当返回值大于等于0时，表示能够获取到同步状态。因此，在共享式获取的自旋过程中，成功获取到同步状
    态并退出自旋的条件就是tryAcquireShared(int arg)方法返回值大于等于0。
    在doAcquireShared(int arg)方法的自旋过程中，如果当前节点的前驱为头节点时，尝试获取同步状态，如果返回值大于等于0，
    表示该次获取同步状态成功并从自旋过程中退出。
    该方法获释放同步锁状态后，将会唤醒后继处于等待的节点，它和独占式的区别是调用tryAcquireShared方法必须保证修改
    同步状态的线程安全，一般是使用CAS来保证的，因为这是释放会出现有多个线程。
4.共享工具类(这是实现一个共享同步锁的一个demo)
    设计一个同步工具：该工具在同一时刻，只允许至多3个线程同时访问，超过3个线程的访问将被阻塞。首先，确定访问模式。
    TrinityLock 能够在同一时刻支持多个线程的访问，这显然是共享式访问，因此，需要使用同步器提供的acquireShared(int args)
    方法等和Shared相关的方法，这就要求TwinsLock必须重写tryAcquireShared(int args)方法和
    tryReleaseShared(int args)方法，这样才能保证同步器的共享式同步状态的获取与释放方法得以执行。
    其次，定义资源数。TrinityLock 在同一时刻允许至多三个线程的同时访问，表明同步资源数为3，
    这样可以设置初始状态 status为3，当一个线程进行获取status减1，该线程释放，则status加1，
    状态的合法范围为 0、1和2,3，其中0表示当前已经有3个线程获取了同步资源，此时再有其他线程对同步状态进行获取，
    该线程只能被阻塞。在同步状态变更时，需要使用 compareAndSet(int expect,int update)方法做原子性保障。
    最后，组合自定义同步器。前面的章节提到，自定义同步组件通过组合自定义同步器来完成同步功能，
    一般情况下自定义同步器会被定义为自定义同步组件的内部类。
5.Condition(做等待通知用的，就是维护了一个等待队列)
    这里是使用了一个等待队列，每一个同步队列可以存在多个等待队列，队列中的节点都包含下一个节点的引用，当一个
    线程调用了Condition.wait()方法，那么该线程将会释放锁，构造节点加入等待队列进入等待状态，其实节点的定义复用了同步器
    中定义的节点的定义，就是说同步队列和等待队列中节点类型都是同步器的静态内部类。
    一个Condition包含一个队列，Condition拥有头结点（firstWaiter）和未节点（lastWaiter），当调用Condition.await()方法，
    将当前线程构造节点，并将节点加入到队列的尾部，Condition拥有首尾节点的引用，而新增节点只需要将原有的尾节点nextWaiter
    指向它，并且更新尾节点即可。上述节点引用更新的过程并没有使用 CAS 保证，原因在于调用await()方法的线程必定是获取了锁
    的线程，也就是说该过程是由锁来保证线程安全的。
    当等待队列中的节点被唤醒，则唤醒节点的线程开始尝试获取同步状态；如果不是通过其他线程调用Condition.signal()方法唤醒，
    而是对等待线程进行中断，则会抛出InterruptedException；意思是不唤醒，而实进行中断，则会抛出中断异常。
    当调用Condition.wait()方法，同步队列的首节点并不会直接加入等待队列，而是通过addConditionWaiter()方法把当前线程构
    造成一个新的节点并将其加入等待队列中。
    调用Condition的signal()方法，将会唤醒在等待队列中等待时间最长的节点（首节点），在唤醒节点之前，会将节点移到同步队列中。
    signal()方法会调用isHeldExclusively()进行检查，检查该线程释放获得了锁，如果获得了锁，则将其移动到同步队列并使用
    LockSupport 唤醒节点中的线程。
    被唤醒后的线程，将从await()方法中的 while循环中退出（isOnSyncQueue(Node node)方法返回true，节点已经在同步队列中），
    进而调用同步器的acquireQueued()方法加入到获取同步状态的竞争中。
    Condition 的 signalAll()方法，相当于对等待队列中的每个节点均执行一次 signal()方法，效果就是将等待队列中所有节点
    全部移动到同步队列中，并唤醒每 个节点的线程。
6.ReentrantLock 的实现
    锁的可重入：在加锁是将state进行加操作，释放锁是对state进行减操作，来实现可重入锁，释放锁进行减操作是使用CAS保证线程安全
    公平和非公平锁：
        非公平锁：lock方法直接使用compareAndSetState(0, 1)获取锁，如果失败再调用acquire(1)方法，该方法又会再次使用
                  tryAcquire再次获取一次，如果还失败，再调用acquireQueued方法将当前线程加入到同步队列中。
        公平锁：tryAcquire方法，该方法与nonfairTryAcquire(int acquires)比较，唯一不同的位置为判断条件多了
                hasQueuedPredecessors()方法，即加入了同步队列中当前节点是否有前驱节点的判断，如果该方法返回true，
                则表示有线程比当前线程更早地请求获取锁，因此 需要等待前驱线程获取并释放锁之后才能继续获取锁。
    非公平锁效率更高，因为cpu在恢复一个被挂起的线程与该线程真正开始运行之间存在着严重的延时问题。
7.读写状态的设计
    读写锁将state变量切分成了两个部分，高16位表示读，低16位表示写，这里是使用的位运算。
    它们之间都是使用的同一个syn所以能实现读写锁变量之间的共享。
8.写锁的获取与释放
    如果获取写锁时，读锁已经获取了，这当前线程等待，该方法除了重入条件（当前线程为获取了写锁的线程）之外，
    增加了一个读锁是否存在的判断。如果存在读锁，则写锁不能被获取，原因在于：读写锁要确保写锁的操作对读锁可见。
    这个可重入锁的实现，和前面的一样，在state变量的低位上做累加来实现锁的可重入，需要使用CAS来保证线程安全。
9.读锁的获取与释放
    读锁是一个支持重进入的共享锁,它能够被多个线程同时获取，在没有其他写线程访问（或者写状态为0）时，读锁总会被成功地获取,
    如果当前线程在获取读锁时，写锁已被其他线程获取，则进入等待状态。
    每个线程各自获取读锁的次数只能选择保存在ThreadLocal中，由线程自身维护。（也就是可重入锁的实现）
    当前线程获取了写锁或者写锁未被获取，则当前线程（线程安全，依靠CAS保证）增加读状态，
    成功获取读锁。读锁的每次释放（线程安全的，可能有多个读线程同时释放读锁）均减少读状态。
10.锁的升降级
    锁降级指的是写锁降级成为读锁。如果当前线程拥有写锁，然后将其释放，最后再获取读锁，这种分段完成的过程不能称之为锁降级。
    锁降级是指把持住（当前拥有的）写锁，再获取到读锁，随后释放（先前拥有的）写锁的过程。
    ReentrantReadWriteLock不支持锁升级（把持读锁、获取写锁，最后释放读锁的过程）。目的是保证数据可见性，
    如果读锁已被多个线程获取，其中任意线程成功获取了写锁并更新了数据，则其更新对其他获取到读锁的线程是不可见的。
