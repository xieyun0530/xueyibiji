1.什么是线程安全
    多线程中访问某个类，不需要加同步或者协作，这个类都能做出正确的行为，那么它就是线程安全的。
2.线程封闭
    就是把一个对象装在一个线程里，这有这一个线程能看到这个对象，这样就算这个对象是线程
    不安全的，也不会出现线程安全问题。
    ad-hoc：
        靠实现者自己实现的。
    栈封闭：
        就是局部变量，也就是在方法中定义的变量，这个是每个线程所独有的，就算多个线程访问，
        那么每个线程也是独立拥有着，因为方法都是存在栈中的，每个线程都拥这个的副本。
    无状态：
        没有任何的成员变量的类，就叫做无状态的类；如果方法的入参有一个对象，那么也是说方法
        入参对象的类是线程不安全。
    让类不可变：
        1.加final关键字，所有的成员变量都加上private，也可以将成员变量加上final，但是如果加了
          这个成员变量是一个对象，那么要保证对象对应的类是不可变的。
        2.不提提供修改成员变量的地方，也不作为方法的返回值；一旦成员变量中有对象，final并不保证
          类的安全，能保证对象的引用不可变，但在多线程情况下，在堆上的实例，可能被多个线程修改。
    volatile：
        不能保证类的安全，只能保证类的可见性，适合一个线程写，多个线程读的情况。
    加锁和cas：
        使用synchronized关键字，使用显示锁，使用原子变量，修改时使用cas机制。
    安全的发布：
        其实意思就是对于类中的成员变量，提供一个get也就是外部能够访问成员变量值的方法，如果是一个
        基本类型的变量，那么发布出去是没有安全问题的，因为基本类型发布出去的其实是这个类的副本，
        而如果是引用类型，通过get发布出去，会造成多线程情况下这个数据的不正确，因为它是持有的对象
        实例的引用，当把这个对象赋值给另外一个新建的对象的时候，这个新建的对象也是跟它指向的是堆中
        同一个对象实例，当被赋值的这个对象修改了以后，对象的实例也就跟着修改了。
        Collections.synchronizedList 进行包装，内部使用了将一个包装类也就是继承了这个类 ，在这个包
        装类中，将传入的这个list的get和set方法使用了synchronized加锁，外部访问的其实是这个子类中加
        了锁的方法，保证了线程安全。
        同样我们也可以使用Collections.synchronizedList中对类进行包装的方法，就是重写一个类，定义一个
        构造函数，入参是被包装类的对象，内部定义一个成员变量，为被包装类的对象，在这个构造函数中使用
        子类创建赋给父类的对象(内部类是子类，被包装的类是父类)，入参是这个被包装的类，也就是说把get和
        set的具体实现交给了子类，也就是这个静态内部类，这个内部类继承被包装的类，并重写get和set方法，
        里面也定义一个被包装类的成员变量，然后在这个get和set方法中使用同一个对象对这个成员变量对象，进行加锁。
        如果这个类是final类型的，不可以继承，那么可以使用委托的形式，其实跟继承的差不多，就是把操作这个类
        的方法，交给另外一个线程安全的类来执行，也就是在这个委托类中进行加锁或者使用其它手段保证线程安全。
    ThreadLock：
        内部实现了线程封闭，里面维护了一个Map，Map的key是每个线程的线程名称，值就是我们要封闭的对象。
        每个线程都对应着一个值，不存在多个线程共享值的情况。
    Servlet辩解：
        线程不安全的类，为什么我们平时感觉不到：
            需求上，很少共享的需求；接受与应答是用同一个线程来做的。
            只有Servlet中有成员变量，在多线程进行写的情况下，就会产生线程安全问题。
    死锁：
        用自己的话来说：假设有两个线程分别是：线程A和线程B，然后又有两把锁分别是：锁C和锁D，在多线程的情况下，
                       线程A抢到了锁C，线程B抢到锁D，但是这个时候线程A和线程B同时都需要锁C和锁D，这个时候
                       就出现了死锁的情况，因为它们都拥有对方的锁，然后又互不相让。
        打破死锁的情况：
            1.这个时候来了一个E锁，拥有C锁同样的功能，那么这个时候线程A获取到了E锁后，就会和B线程发生争夺了。
            2.现在如果规定，线程A和线程B，都只能先获取C锁再获取B锁，这样就不会出现两个线程分别拥有C锁和D锁的情况。
        总结：
            死锁发生在多个操作者(m>=2)，争夺多个资源的情况下(n>=2),且 (N<=M）才会发生这种情况；单线程不发生死锁。
        学术的定义：
            互斥条件：就是一段时间内，这个资源只能被一个线程占用。
            请求与保持：就是获取到一个锁，然后又去获取另一个锁，但是获取到的这个锁不会释放。
            不剥夺条件：就是获取到了一个锁，只有自己使用完了，才会释放，没有使用完之前不会释放。
            环路条件：就是说线程A正在等线程B的锁，而线程B又在等线程A的锁，形成了互相等待的环形条件。
        现象：
            简单死锁：
                就是锁的顺序相反，A方法先获取锁C再获取锁D，而B方法先获取锁D再获取锁C。
            动态死锁：
                就是A对象的入参是先入参锁C，锁D，而B对象则先入参锁D，锁C，两个完全相反。
        解决：
            定位：
                进入jdk的bin目录，通过 jps 查询应用的 id，再通过 jstack id 查看应用的锁的持有情况。
            修正：
                锁的顺序一致：
                    1.内部通过比较确定拿锁的顺序，比如比较两个值得hash。
                    2.采用尝试拿锁，也就是使用ReentrantLock的tryLock超时拿锁机制，finally块的释放锁。
    活锁：
        A线程和B线程在尝试拿锁的过程中，A线程获取了C锁，B线程获取了D锁，然后A线程在尝试获取D锁，而B线程
        尝试获取C锁，但是发现都获取不到，从而释放自己已经拿到的锁，一直重复着这样的操作，这样的过程就是
        产生了活锁。
        也就是说获取锁的条件没有被满足，一直尝试获取和释放，重复着这样的操作。
        解决办法：每个线程休眠一个随机数，错开拿锁的时间。
    线程饥饿：
        低优先级的线程，总是拿到执行时间。
    并发下的性能：
        首先保程序能够正确的执行，达不到要求的时候再进行优化。
        是否优化，要以测试为准。
    线程引入的开销：
        上下文切换：
            就是说多线程情况下，CPU要将线程A挂起，来执行线程B的过程。
            这个切换时间大概是50到10000个始终周期，也就是几微妙。
            unix系统可以使用vmstat命令查看上下文切换次数参数是cs。
    内存同步开销：
        同步的性能开销包括多个方面，在synchronized和volatile提供的可见性中，使用了内存屏障，内存屏障可以
        刷新内存，是缓存无效刷新，使硬件来写缓存，以及停止执行管道，所以对性能有影响，还会在大多数的情况下
        禁止重排序。
    阻塞：
        阻塞包括上下文切换，一次是阻塞的时候，一次是唤醒的时候。
    如何减少锁的竞争：
        减少锁的粒度：
            也就是只锁业务逻辑的部分代码，而不是全部。
        减少锁的范围：
            就是说，尽量缩小持有锁的时间，将与锁无关的代码放到锁的外面
        避免多余的锁：
            就是说两个锁之间的代码执行时间非常短，可以合并成一个锁，避免多次加锁释放锁，因为加锁和释放锁
            是会影响性能的。
        分段锁：
            concurrentHashMap就是典型的分段锁。
        替换独占锁：
            在业务允许的情况下：
               1.使用读写锁
               2.使用自旋CAS
               3.使用系统的安全并发容器
3单例模式
    懒汉模式：
        类初始化模式，也叫延迟占位模式。在单例类的内部由一个私有静态内部类，来持有这个单例类的实例。
        延迟占位模式还可以用在多线程下实例域的延迟赋值。
    双重检查锁的方式：
    public class Singleton{
        //单例引用volatile修饰，1.禁止指令重排序，2.保证变量可见性
        private static volatile Singleton singleton;

        public static Singleton getInstance(){
            if(singleton == null){                      //1.第一次检查
                  synchronized(Singleton.class){        //2.在Singleton类对象上加锁
                        if(singleton == null){          //3.第二次判断为空
                            singleton = new Singleton();//4.实例化
                        }
                  }
            }
        }
    }
    1.为什么Singleton对象要同时使用volatile修饰和synchronized锁？
       因为创建Singleton对象(singleton = new Singleton())分为三步：
        1.申请一块内存空间
        2.初始化申请好的内存空间
        3.将内存地址的引用指向singleton对象
    为什么使用volatile：
     使用volatile修饰是为了禁止指令重排序，也就是保证创建对象的3步骤，能够有序的执行；如果只使用synchronize锁，
     如果当线程A执行到第一步或者第二步的时候，发生了线程切换，B线程去访问singleton对象的时候会出现空值。
   为什么要使用了volatile还要使用synchronized？
     如果执行到第一步或者第二步，发生了线程切换，这个时候singleton的对象还是空值，如果没有加synchronized锁，
     则第二个线程执行if(singleton == null)这个条件的时候，会发现还是一个空的，就会又进去创建，这样会有两个线程去
     创建了这个对象。
   2.第一次检查了singleton为空为什么还要第二次进行检查？
    当线程A执行到singleton创建对象的第一步或者第二部的时候，发生了线程切换，线程B进入第一个if(singleton == null)
    发现singleton对象为空，这个时候B线程碰到synchronized会阻塞，但是当A线程执行完了以后，B会串行的方式，再次进入
    synchronized代码块，这个时候里面没有对这个对象判处为null，则B线程会再将singleton对象重新再创建一次。
