1.索引是帮助MySql高效获取排好序的数据结构
    索引的数据结构
    二叉树 ----因为极端情况下会形成链表
    红黑树 ----数据量非常大的情况下会，树的深度太深
    B-Tree ：
        叶子节点具有相同的深度，叶子节点的指针为空；
        所有索引元素不重复；
        节点中的数据索引从左到右递增排序；
        将一个节点储存多个元素，每个元素是key-value进行存储，key存放索引，value存放指向磁盘的数据指针，这样树的高度就变低了；
    B+Tree：
        非叶子节点不存储data，只存储索引（冗余），这样可以放更多的索引；
        叶子节点包含所有索引字段；也就是key-value的形式，key存放索引，value存储指向磁盘的数据指针；
        叶子节点用指针连接(双向指针)，提高区间访问能力；---在范围查询的时候，顺着这个指针去找到相邻的数据
        非叶子节点中储存的是从小到大的数据索引，索引中间存储指向子节点的指针；索引是Integer类型那么一个索引就是8b，指针是6b；
        mysql中B+Tree节点是16k；
        那么一个非叶子节点能存多少个索引?
            16k/(8b+6b)大概是等于1170个
        假设叶子节点是的数据加索引是1kb，那么一个高度为3的B+Tree最多能存多少数据？
            1170*1170*16大概等于2千多万
    mysql索引的运行原理：
        mysql首先IO将整个节点的索引都加载到内存中进行对比，对比后找到指向下一个节点的指针后，根据指针IO到下一个节点把它
        再加载到内存中进行比较，循环次操作，直到找到这个数据所在的索引位置，也就是叶子节点，把整个叶子节点加载到内存中，
        找到索引的位置，因为叶子节点的索引是对应的真实数据的指针，也就是key-value的形式，取出value中的指针，然后根据指针IO到真实的数据。
    mysql中的根节点是常驻在内存中的；
    like 'dafei%'是否走索引：
        这里根据索引匹配原则，是会走索引，但是如果表中的数据都是dafei开头的数据，就会不走索引；
    回表操作：
        因为非主键索引，也就是二级索引，
    引擎是基于表的；
2.Myisam存储引擎
    MYISAM储存引擎有三个文件
    .frm结尾的文件  -----存储数据结构
    .myd结尾的文件 ------存储表中的数据
    .myi结尾的文件 ------储存索引的文件
    mysql查询步骤，首先看查询条件是否是索引字段，根据索引字段到.myi的B+Tree中快速的找到这个元素，从根节点开始查找，找到后取出data中的元素，
    data元素存的是数据的地址，根据这个地址到.myd文件中把这个数据,最后把这个数据load到内存中
    叶子节点中的data存储的是数据指针地址；
3.InnerDB存储引擎
    InnerDB储存引擎有两个文件
    .frm结尾的文件  -----存储数据结构
    .ibd结尾的文件 ------表数据和索引的合成文件，索引跟数据存储在一起
    叶子节点中的data储存的是索引所对应的字段数据
4.聚集索引
    只有主键索引才是聚集索引，因为只有主键索引中存储了数据行的指针；而二级索引的叶子节点存储的是主键和非主键数据，如果查询
    的时候没有使用主键，而是使用的非主键查询且查询字段没有包含在索引字段，则会进行回表查询，就是根据普通索引先找到索引中
    存储的主键，然后根据主键再去查询到数据行；
    索引优化器：比如有一个联合索引，A，B，C，使用where B = 'aaa' and A = 'sdfasdf'查询是，索引优化器会将索引优化成
                最左匹配原则；
5.面试题
    (1)为什么innoDB表必须有主键，并且推荐使用整形的自增列？
        如果设置了主键，那么InnoDB会选择主键作为聚集索引、如果没有显式定义主键，则InnoDB会选择第一个不包含有NULL值的唯一索引作为
        主键索引、如果也没有这样的唯一索引，则InnoDB会选择内置6字节长的ROWID作为隐含的聚集索引(ROWID随着行记录的写入而主键递增)。
    (2)为什么非主键索引结构叶子节点储存的是主键值？
        1.减少了出现行移动或者数据页分裂时二级索引的维护工作（当数据需要更新的时候，二级索引不需要修改，只需要修改聚簇索引，
            一个表只能有一个聚簇索引，其他的都是二级索引，这样只需要修改聚簇索引就可以了，不需要重新构建二级索引）；
        2.聚簇索引也称为主键索引，其索引树的叶子节点中存的是整行数据，表中行的物理顺序与键值的逻辑（索引）顺序相同。
            一个表只能包含一个聚集索引。因为索引（目录）只能按照一种方法进行排序。
        3.非聚簇索引（普通索引）的叶子节点内容是主键的值。在InnoDB里，非主键索引也被称为二级索引（secondary index）。
    (3)联合索引
        联合索引，是一个最左匹配原则，遇到范围查询后，后面的字段就会失效；
    (4)为什么MySQL的索引要使用B+树而不是其它树形结构？比如B树？
        因为B树不管叶子节点还是非叶子节点，都会保存数据，这样导致在非叶子节点中能保存的指针数量变少（有些资料也称为扇出），
        指针少的情况下要保存大量数据，只能增加树的高度，导致IO操作变多，查询性能变低；
    (5)mysql没有声明主键索引的危害？
        1.在没有索引的情况下会将行锁升级成表锁；
        2.在使用自增列的时候，如果没有将这个列声明成主键，当自增键到达int类型的最大值2^32-1的时候，会从0开始，并且会将之前
          的自增值进行覆盖，而使用了主键则会包主键冲突的异常；
        3.如果没有主键，mysql会默认创建一个_rowid来做主键索引，而这个_rowId主键索引会占据6Byte，而int则只会占用4Byte，
           这样造成了资源浪费；
    (6)间隙所，临界锁，自增锁
        间隙锁和临界锁，锁数据的时候不会包含前面这个数据，但是会包含后面这个数字；
    (7)MVCC使用快照来实现间隙锁
        实现逻辑就是，每个事物都有一个transaction_id，而mvcc则会有记录一个up_limit_id，这个id是记录最后一个事务执行完成的
        事务id；也就是快照读；
        可重复读（repeatable-read）：MySql在一定程度上解决了不可重复读问题，但是某些情况下是没有的，也就是当前读的情况下，
            当一个事务需要去修改一个数据的时候，会先去读取数据库中已经提交的事务的真实数据，而不是读取快照数据，虽然这种
            情况产生了不可重复读现象，但是如果不触发当前读依然使用快照读就会出现，导致数据不一致的情况，所以说mysql的
            innerDB引擎对于这种情况的处理是一个正确的处理方式；
        快照读：读取的数据是up_limit_id对应的数据；
                一个事务执行的时候有三个id分别是：
                            trx_id：事务开启时innerDB分配的事务id，
                            up_limit_id：当前最后事务的版本号；
                            row_txr_id：数据版本号；
        当前读：读取的是当前数据库的真实数据；
        什么时候触发快照读，什么时候触发当前读？
            例如：
                事务A读取前，假设up_limit_id = 999；当事务A读取数据的时候，三个id分别是：trx_id = 1000，up_limit_id=999，
                row_txr_id = 1000；
                事务B读取，三个id分别是：trx_id = 1001，up_limit_id=999，row_txr_id = 1001；
                事务C读取，三个id分别是：trx_id = 1002，up_limit_id=999，row_txr_id = 1003；
            执行sql：
                事务A执行两条一样的sql查询；
                事务B执行一条sql查询，一条update操作，一条sql查询；
                事务C执行一条update操作；
                执行顺序：事务A执行第一次查询，事务B执行第一次查询，事务C执行update；事务B执行update再执行查询，
                           事务A再执行一次查询；
            执行结果：
                事务A的两个查询数据会使一样的，因为A事务是触发的快照读操作；
                事务B的两次查询不一样，因为当事务B执行update时会触发当前读操作，则就会读取到事务C修改后的值，最终导致
                两个查询的结果不一样；

6.explain关键字分析
    (1)explain关键字列表组成
        id、select_type、table、type、possible_keys、key key_len、ref、rows、Extra
        1.id：表示查询的序列号；
        2.select_type：用于区别:普通查询(simple)、联合查询(UNION RESULT与UNION)、子查询等的复杂查询(primary与subquery);
            在FROM列表中包含的子查询被标记为derived(衍生);
        3.type：system>const>eq_ref>ref>range>index>ALL；一般来说，得保证查询至少达到range 级别，最好能达到ref。
            System与const：表示表中只有一条记录；
            eq_ref： 唯一索引扫描，表中只有一条记录进行匹配；
            Ref：非唯一性索引扫描，返回匹配某个单独值的所有行；
            Range：值扫描给定范围的索引，比如在where语句中使用了between、<、>、in 等的查询；
            index：全索引扫描，只是扫描了所有得索引而没有去查询数据；
            all：遍历全表找到匹配行；
        4.possible_keys：可能用到的key；
        5.key：实际用到的索引，如果为null，则表示没有用到索引；
        6.key_len：表示使用的字节数；通过索引列的类型计算出索引的长度，在能装得下数据的前提下，越小越好；
            字符编码根据不同的编码占用的空间不同：latin：1字节，gbk：2字节，utf8：3字节；
            字符类型char类型时 = char(10) * 3 + (可以为null) 1；
            字符类型varchar类型时 = varchar(10) * 3 + (可变字符)2 + (可以为null) 1；
            整数/浮点数/时间类型的索引长度：NOT NULL=字段本身的字段长度；
                NULL=字段本身的字段长度+1(因为需要有是否为空的标记，这个标记需要占用1个字节)；
        7.ref：显示索引的那一列被使用到了;就是哪些列或者常量被索引用于查找了数据；
        8.rows：根据表统计和索引使用情况，大概估算出找到所需要的记录需要读取多少行数据；
        9.extra：显示索引使用情况详情；这里有以下值：
            Using filesort：使用了一个外部索引排序，而不是表内索引顺序进行读取；(可以优化)
            Using temporary：使用了临时表保存中间结果，MySql对数据进行排序是常用临时表，常见于order by,group by;(上面和这个同时出现要注意)
            Using index：使用了覆盖索引，避免了访问表的数据，效率不错；
                如果同时出现了using where表示索引被用来执行索引键值的查找；如果没有出现发，表示索引用来读取数据非执行查找动作；
                覆盖索引：使用select查询的时候，select中的字段都是索引字段，也就是通过索引就能读取到所有数据，而不必再读取数据文件，
                    但是不能将表中的数据都建立索引来达到覆盖索引的目的，这样会加大索引的维护难度；
            Using where：表名使用了where过滤；
            using join buffer：表明使用了连接缓存；
            impossible where：where子句的值总是false，不能获取到任何值；
7.SQL优化
    (1)尽量全量匹配
        当建立了索引列后，能在where 条件中使用索引的尽量所用。
    (2)最佳左侧匹配原则
    (3)不在索引列上做任何操作
    (4)范围查询放最后
        中间有范围查询会导致后面的索引失效
    (5)覆盖所有尽量用
        只查询出有索引的字段，且查询的顺序于建立索引的顺序一致；
    (6)不等于要慎用
        如果要使用不等于，请使用覆盖索引；
    (7)is null/is not null有影响
        当字段没有定义is not null时：使用is null会走索引，使用is not null不会走索引；
        当字段定义了is not null时：使用is null或者is not null都会导致索引失效；解决方式使用覆盖索引；
    (8)like查询要当心
        左匹配会使索引失效，解决办法使用覆盖索引；
    (9)字符类型加符号
        字符类型查询时需要加引号；
    (10)or改union效率高
        or查询不走索引的解决方式使用覆盖索引；
    (11)in子查询会走索引，如果in的区间大多就会全表，不同版本阈值不同，5.7以后好像是200多个；not in不会走索引；
8.慢查询
    (1)慢查询基本配置
        1.slow_query_log 启动停止技术慢查询日志；set GLOBAL slow_query_log = 1;开启慢查询；
        2.slow_query_log_file 指定慢日志查询的存储路径(默认和数据文件放在一起)；
        3.long_query_time 指定慢查询日志SQL执行时间的阈值(单位：秒，默认10秒)；set global long_query_time=10;
        4.log_queries_not_using_indexes 是否记录未使用索引的SQL；
        5.log_output 日志存放的地方【TABLE】【FILE】【FILE,TABLE】，set global log_output='FILE,TABLE'项目开发中日志
            只能记录在日志文件中，不能记表中；
    (2)慢查询分析工具
        Mysqldumpslow：语法 mysqldumpslow -s r -t 10 slow-mysql.log；
        pt_query_digest：这个工具能将sql的执行数据进行统计展示，结果更直观；
