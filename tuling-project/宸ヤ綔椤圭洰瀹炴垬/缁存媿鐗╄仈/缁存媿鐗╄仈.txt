4.db.orderSystem_distributionModel.aggregate([
      {
          $match: {
              insertTime: {
                  $gte: ISODate("2000-02-13T16:00:00Z"),
                  $lte: ISODate("2026-03-13T19:46:32.999Z")
              }
          }
      },
      { $limit: 10000 }
  ])查询的数据缺少一条insertTime时间等于：2023-03-13 03:46:31.973的数据
    原因：MongoDB支持弱类型存储，即同一个字段（如insertTime）可以存储不同的数据类型（Date、String、NumberLong等）。
        但查询时是严格类型匹配的，也就是说：
        如果insertTime存的是Date类型，find查询时用Date类型才会匹配到；
        如果insertTime存的是String类型，用Date类型查询不会匹配到，而必须用String查询。
        分组统计这个字段的类型有那几种：
        db.orderSystem_distributionModel.aggregate([
            {
                $group: {
                    _id: { $type: "$insertTime" }, // 按类型分组
                    count: { $sum: 1 }             // 统计每种类型的数量
                }
            }
        ])
    解决方法：直接使用String类型去匹配查询，就会匹配类型时String类型的数据
        db.orderSystem_distributionModel.find({
            insertTime: {
                $gte: "2000-02-13T16:00:00Z",
                $lte: "2026-03-13T19:46:32.999Z"
            }
        })

7.异步导出任务，采用rabbitMQ实现
    1.只创建一个队列且只有一个消费者组，也就是一个消费者进行串行化
    2.生成导出数据的时候加一个分布式锁，只允许有一个线程进行生成任务，设置消费者同步回复ack，没消费完成一直阻塞
    3.针对生成导出数据耗时较长的任务，导致ack超时的问题，也就是消费超时，会导致生产者重试，那我们执行任务时先判断一下分布式锁是否存在
        如果存在则使用sleep阻塞该线程一段时间后，再次获取分布式锁任务，如果还是获取不到则把该消息再次发送到队列里面，然后结束回复ack，
        这里下一个任务过来还是走第三步逻辑重复操作，这样超时的消息会一直往后面再次排队，解决了消费者没有消费成功的问题；
    4.如果获取到分布式锁了，则先判断一下生成任务的状态，如果是已完成则不重复生成了，直接结束方法执行完成，否则才走生成任务的逻辑；
    5.如果还未生成完成服务重启了，生产者会进行重试；

15.使用mybatisPlus的批量新增语句，出现死锁
    原因：每次批量新增1000条，导致间隙锁死锁，因为这1000条的数据是多个insert而不是insert values，查看死锁持有的索引，因为锁是基于索引的
        查看后UNIQUE KEY `idx_owner` (`owner`,`date`,`type`)两个线程死锁线程持有的锁是这个索引
    解决办法：因为新增时owner是无序的，导致新增事务的时候产生了间隙，所以导致死锁，解决办法把批量新增改成100要间隙变小，或者按照owner排序；

    的过滤器的isAccessAllowed()执行了两次
    原因是：@Bean("shiroFilterFactoryBean")
          public ShiroFilterFactoryBean shiroFilter(SecurityManager securityManager){
            // 添加自己的过滤器并且取名为jwt
            Map<String, Filter> filterMap = new HashMap<String, Filter>(1);
            //如果cloudServer为空 则说明是单体 需要加载跨域配置【微服务跨域切换】

            shiroFilterFactoryBean.setFilters(filterMap);
          }
          @Bean
          public FilterRegistrationBean shiroFilterRegistration() {}
        注入了两次，导致过滤器被执行两次，会导致如果过滤器中取拿body里面的请求参数时会在第二次那的时候拿不到，因为请求留只能拿一次

    解决办法：
        1.方法一：可以删除shiroFilterRegistration()这个Bean的注入
        2.方法二：新增一个过滤器且顺序放到最前面：
        @Bean
            public FilterRegistrationBean<WebFilter> webFilterRegistration(WebFilter webFilter) {
                FilterRegistrationBean<WebFilter> registration = new FilterRegistrationBean<>();
                registration.setFilter(webFilter);
                registration.setOrder(Integer.MIN_VALUE); // order 越小越优先
                registration.addUrlPatterns("/*");
                return registration;
            }
        @Slf4j
        @Component
        public class WebFilter implements Filter {

            public void doFilter(ServletRequest servletRequest, ServletResponse servletResponse, FilterChain filterChain){
                servletRequest = requestWrapper(servletRequest);
                filterChain.doFilter(servletRequest, servletResponse);
            }
            /**
             * 请求包装(特定条件下进行请求包装,支持body重复读取)
             *
             * @param servletRequest 原请求
             * @return 包装后的请求
             */
            public ServletRequest requestWrapper(ServletRequest servletRequest) {
                if (servletRequest instanceof HttpServletRequest) {
                    HttpServletRequest req = (HttpServletRequest) servletRequest;
                    String contentType = req.getContentType();
                    if (contentType == null) {
                        contentType = "";
                    }
                    // POST请求类型，并且不是表单请求,才获取POST请求体
                    if (CommonConstant.HTTP_POST.equals(req.getMethod())
                            && !contentType.toLowerCase().contains("form")) {
                        return new BodyReaderHttpServletRequestWrapper(req);
                    }
                }
                return servletRequest;
            }
    	}

                <id>spring-snapshots</id>
                <name>Spring Snapshots</name>
                <url>https://repo.spring.io/snapshot</url>

            例如：jeecg的xx-api-1.0.0-SNAPSHOT.jar可能在spring snapshot、jeecg snapshot、sonatype snapshot中都存在，
            但版本内容不同，先匹配到的就被使用了，不再尝试后面的。

26.Caused by: java.lang.NoClassDefFoundError: com/obs/services/model/AccessControlList
    原因：这个类是通过下面的pom引入的，而<scope>provided</scope>这个标签的意思是依赖不会往下传递导致的
        <dependency>
            <groupId>com.huaweicloud</groupId>
            <artifactId>esdk-obs-java</artifactId>
            <version>3.21.8</version>
            <scope>provided</scope>
        </dependency>
    解决办法：把<scope>provided</scope>注释掉就行了
